5.1 Why Protocols and Reporting Standards Matter

Real-world evidence (RWE) studies face a credibility challenge that randomized trials do not: every analytic decision—from cohort definition to confounder selection to model specification—is made by the analyst after seeing the data. Without pre-specification and transparent reporting, it is impossible for reviewers, regulators, or replication teams to distinguish principled analysis from data dredging.

The causal roadmap introduced in Chapter 1.2 provides the scientific backbone: define the question, specify the causal model, identify assumptions, choose an estimator, and evaluate results. This chapter provides the documentation backbone—the templates, checklists, and logs that make each roadmap step auditable and reproducible.

Three complementary frameworks have emerged to fill this gap:

  • STaRT-RWE (Wang et al. 2021): a structured template for planning and reporting RWE studies
  • RECORD (Benchimol et al. 2015): an extension of STROBE for studies using routinely collected health data
  • TARGET (Cashin et al. 2025): a statement specifically for target trial emulation studies

Each serves a different purpose, and a well-documented RWE study will draw on elements from all three.

Alignment Is Everything

The most common error in target trial emulation is misalignment of time zero with eligibility and treatment assignment. If these three elements are not synchronized, immortal time bias results. The TARGET template forces you to make this alignment explicit.

5.6 Regulatory Auditability

What Regulators Expect

Regulatory agencies including FDA (U.S. Food and Drug Administration 2018, 2023a, 2024), EMA, and NICE (National Institute for Health and Care Excellence 2022) increasingly accept RWE for regulatory decisions—but only when the evidence meets high standards of transparency and reproducibility. The following elements must be logged and available for audit:

  1. Data access dates: When was the data extracted? From which version of the database?
  2. Cohort selection criteria: Exact inclusion/exclusion logic with code lists
  3. Variable definitions: Operational definitions for every variable, including lookback windows and code algorithms
  4. Model libraries with versions: Exact package versions (e.g., tmle 1.5.0-2, SuperLearner 2.0-28)
  5. Random seeds: Every call to a random number generator must use a documented seed
  6. Code hash: SHA-256 hash of the final analysis script(s)
  7. Analyst identity: Who ran the analysis, with institutional affiliation
  8. ROBINS-I assessment: Risk of bias evaluation using the ROBINS-I tool (Sterne et al. 2016)

A practical approach is to generate this metadata automatically at the end of your analysis script:

# Auditability metadata block
audit_log <- list(
  analyst       = Sys.info()["user"],
  date_run      = Sys.time(),
  R_version     = R.version.string,
  platform      = .Platform$OS.type,
  packages      = renv::status()$library$Packages,
  random_seed   = .Random.seed[1:5],
  script_hash   = digest::digest(file = "analysis.R", algo = "sha256")
)
saveRDS(audit_log, "audit_log.rds")

5.8 Check Your Understanding

Questions

  1. Protocol locking: You discover after data extraction that your pre-specified exposure definition captures only 60% of treated patients due to a coding change in the database. What should you do, and how should you document this?

  2. TARGET alignment: A colleague’s target trial emulation defines eligibility as “patients with a diagnosis of heart failure in 2020” but defines time zero as “first prescription of the study drug.” Explain why this creates immortal time bias and how the TARGET template would have caught it.

  3. Reproducibility: Your analysis uses SuperLearner with SL.glmnet and SL.ranger. Six months later, both packages have been updated. What components of the reproducibility bundle protect against version-related result changes?

Answers

  1. Record the deviation in the deviation log (Table 5.4). Update the exposure definition with full transparency, noting the original specification, the new specification, and the rationale (coding change). Run the analysis under both definitions as a sensitivity analysis and report both results. The key principle is that no change should be invisible.

  2. Patients must survive and remain untreated from their heart failure diagnosis until they receive the study drug. This disease-to-treatment gap creates immortal time—a period during which death (the outcome) cannot occur by definition. The TARGET template forces alignment by requiring that eligibility, treatment assignment, and follow-up start (time zero) be defined simultaneously. If the template is filled correctly, the misalignment becomes immediately visible.

  3. The renv.lock file records exact package versions and can restore them via renv::restore(). The Docker container provides an additional layer by freezing the entire R environment, including system libraries. Together, these ensure that the same code produces the same results regardless of when it is rerun.

## STaRT-RWE: Structured Template for Planning and Reporting {#sec-start-rwe}

The STaRT-RWE framework [@wang2021start] organizes an RWE protocol into a series of structured elements. The table below can serve as a fillable template for your own studies.

| Element | Description | Your Study |
|---------|-------------|------------|
| **Research question** | Specific causal or descriptive question in PICO(T) format | |
| **Data source(s)** | Name, type (EHR, claims, registry), geographic coverage, time span, data access dates | |
| **Study design** | Cohort, case-control, self-controlled, target trial emulation, etc. | |
| **Study population** | Inclusion/exclusion criteria with operational definitions and codes (ICD, CPT, ATC) | |
| **Exposure definition** | Drug, procedure, or intervention; how ascertained; grace periods; new-user vs prevalent-user | |
| **Comparator** | Active comparator, no treatment, or standard of care; rationale for choice | |
| **Outcome definition** | Primary and secondary outcomes; ascertainment method; validation status | |
| **Time zero (index date)** | How the start of follow-up is defined; alignment with eligibility and exposure | |
| **Follow-up** | Duration, end-of-study date, censoring rules | |
| **Confounders** | List of measured confounders with operational definitions; baseline vs time-varying | |
| **Statistical methods** | Estimator (TMLE, IPTW, g-computation, etc.), nuisance model specification, Super Learner library | |
| **Missing data** | Strategy for missingness (complete case, imputation, bounded analysis) | |
| **Sensitivity analyses** | Planned sensitivity analyses (unmeasured confounding, alternative definitions, subgroups) | |
| **Sample size / power** | Expected sample size; power considerations if applicable | |

## TARGET Statement for Target Trial Emulation {#sec-target}

When your RWE study is framed as a target trial emulation [@hernan2016target], the TARGET statement [@cashin2025target] provides a structured way to specify the trial you are trying to emulate and how each protocol element maps to available data.

| Protocol Element | Target Trial Specification | Emulation Using RWD |
|-----------------|---------------------------|---------------------|
| **Eligibility criteria** | Who would be eligible for the trial? | Operational definitions using codes and dates |
| **Treatment strategies** | What interventions are being compared? | Exposure definitions and comparator logic |
| **Treatment assignment** | How would participants be assigned? | Natural assignment mechanism; justify exchangeability |
| **Follow-up period** | Start (time zero), end, duration | Index date definition; censoring rules |
| **Outcome** | Primary outcome and how measured | Outcome ascertainment codes and validation |
| **Causal contrast** | ITT, per-protocol, or other estimand | Map to ICH E9(R1) estimand framework |
| **Analysis plan** | Statistical approach for the target trial | Estimator choice (TMLE, IPTW); nuisance models |

|--------------------|--------------------|-------------------|
| **1. Define the question** | PICO(T) statement; target trial protocol | STaRT-RWE (Research question); TARGET |
